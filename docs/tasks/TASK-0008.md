# TASK-0008: Application Workflow — Session-Aware UI

This task enforces a strict session-based workflow across the entire UI. The concept of a **session** becomes the central axis around which all user interactions are gated. It also introduces a new "add single occurrence" feature available only while a session is active.

---

## Goal

Make the application aware of whether a session is running or not, and use that state to:

1. Gate the batch input: show it only when **no** session is active; hide it entirely once a session starts.
2. Show a compact single-occurrence input **only** while a session is active, accepting only occurrences that are strictly after the last persisted record.
3. Ensure every addition (batch or single) immediately re-renders all sections (occurrences, statistics, prediction).

---

## Definitions

- **Session active**: `loadRecords()` returns at least one record.
- **Session inactive**: `loadRecords()` returns an empty array.
- **Last record**: the chronologically last ISO string in the array returned by `loadRecords()`, which is always kept sorted by `dataService.js`.

---

## New Decision: DEC-0003

Before beginning implementation, append the following entry to `docs/decisions.md`:

```
## DEC-0003 — Session-Aware Workflow (2026-04-30)

**Decision**: The UI is always in one of two mutually exclusive modes:

- **No-session mode**: The batch textarea + submit button are visible. All other
  sections (occurrences, statistics, prediction, single-add) are hidden.
- **Active-session mode**: The batch textarea is hidden entirely. The occurrences,
  statistics, and prediction sections are shown (subject to their own data guards).
  A compact "add single occurrence" field is shown.

**Session definition**: a session is active when `loadRecords()` returns a non-empty
array. There is no separate session flag; the presence of data is the session.

**Single-occurrence rule**: while a session is active the user may add one occurrence
at a time. The submitted value must parse as a valid, unambiguous datetime and must be
strictly after the last persisted record. If either condition fails the input is rejected
with a descriptive error message and nothing is persisted.

**Rationale**: Separating the two modes prevents the ambiguity of half-populated
sessions, avoids mixed-format accidents on append, and makes the intended usage flow
self-evident from the UI state alone.
```

## Steps

### 1. `dataService.js` — expose `getLastRecord()`

Add and export a helper:

```js
/**
 * Returns the ISO string of the most recent persisted record, or null if the
 * store is empty.
 *
 * @returns {string|null}
 */
export function getLastRecord() {
  const records = loadRecords();
  return records.length > 0 ? records[records.length - 1] : null;
}
```

Because `loadRecords()` always returns a chronologically sorted array, the last element is always the newest record.

### 3. `uiController.js` — session-aware render orchestration

#### 3a. Import `getLastRecord`

```js
import { loadRecords, addRecords, clearRecords, getLastRecord } from './dataService.js';
```

#### 3b. `applySessionMode(active)`

Introduce a single function that switches the entire UI between the two modes:

```js
function applySessionMode(active) {
  // batch input: visible only in no-session mode
  const inputSection = getEl('input-section');
  if (inputSection) inputSection.classList.toggle('hidden', active);

  // single-add: visible only in active-session mode
  const singleSection = getEl('single-add-section');
  if (singleSection) singleSection.classList.toggle('hidden', !active);
}
```

This function must be called at the end of every operation that can change the session state: `renderList()`, the New Session confirm handler, and `initUI()` (on page load).

#### 3c. Update `renderList()`

`renderList()` must call `applySessionMode(records.length > 0)` **before** returning in all code paths (both empty and non-empty).

```js
export function renderList() {
  const listEl    = getEl('occurrence-list');
  const sectionEl = getEl('occurrences-section');
  if (!listEl || !sectionEl) return;

  const records = loadRecords();
  applySessionMode(records.length > 0);

  if (records.length === 0) {
    listEl.innerHTML = '';
    sectionEl.classList.add('hidden');
    renderStatistics();
    renderPrediction();
    return;
  }

  listEl.innerHTML = records.map(buildCardHTML).join('');
  sectionEl.classList.remove('hidden');
  renderStatistics();
  renderPrediction();
}
```

#### 3d. Update the New Session confirm handler

After `clearRecords()` and `renderList()` (which already calls `applySessionMode`), also clear the single-add input and its feedback:

```js
clearRecords();
const inputEl = getEl('occurrence-input');
if (inputEl) inputEl.value = '';
clearFeedback();

const singleEl = getEl('single-occurrence-input');
if (singleEl) singleEl.value = '';
setSingleFeedback('', '');

renderList();               // calls applySessionMode(false) internally
renderStatistics();
renderPrediction();
```

#### 3e. `initSingleAddSection()` — new handler

Wire up the single-add input and button. This function is called from `initUI()`.

**Parsing rules for single add:**
1. Pass the trimmed input text through `parseOccurrences()` (reuse the existing parser).
2. Expect exactly one valid token (`valid.length === 1`) and `homogeneous === true`.
3. If `valid.length !== 1`, surface an error: `"Please enter exactly one occurrence datetime."`
4. Retrieve `getLastRecord()`. If the parsed date's ISO string is **not strictly greater than** the last record's ISO string, surface: `"The new occurrence must be after the last recorded occurrence (${formatted last date})."`  — format the last record date using `formatOccurrenceDate` + `formatOccurrenceTime` for readability.
5. On success: call `addRecords([isoString])`, clear the input, set a success feedback, and call `renderList()`.

Feedback for the single-add section must use its **own** `<p id="single-add-feedback">` element, not `#input-feedback`. Introduce helpers `setSingleFeedback(message, type)` and `clearSingleFeedback()` mirroring the existing `setFeedback` / `clearFeedback` pair.

Pressing **Enter** in the single-add `<input>` must trigger the same logic (no Shift check needed for a single-line input).
