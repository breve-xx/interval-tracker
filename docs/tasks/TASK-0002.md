# TASK-0002: Occurrence Input, Validation, and Persistence

This task implements the data entry layer of the application: a freeform text input that accepts a list of event occurrences, validates and normalises them into datetime records, persists them to `localStorage`, and renders a rudimentary confirmation list. The definitive display UI and its styling are deferred to a future task.

## Goal

Allow users to paste an arbitrary block of text containing one or more event occurrence timestamps. The system must parse the text flexibly (tolerating various common datetime formats), extract date-and-time pairs, persist them, and show a minimal confirmation of what was stored.

---

## Steps

### 1. HTML — Input Area (`src/index.html`)

Add the following structural elements inside `#app-container`:

- A `<textarea id="occurrence-input">` where the user pastes their text. Placeholder text should hint at the accepted formats (e.g. `"2024-03-15 14:30:00"`).
- A `<button id="submit-occurrences">` that triggers parsing and persistence.
- A `<div id="occurrence-list">` as the rudimentary display area for persisted records (to be replaced in a future task).
- A `<p id="input-feedback">` element reserved for inline validation messages (errors or success confirmations).

### 2. CSS — Minimal Input Styling (`src/css/styles.css`)

Add only the structural rules needed for usability; aesthetic polish is deferred:

- `#occurrence-input`: `width: 100%`, a reasonable `min-height` (e.g. `120px`), `resize: vertical`.
- `#input-feedback`: distinct colour for error state (`.error`) vs. success state (`.success`) using CSS classes.
- `#occurrence-list`: basic list reset; each entry displayed as a block row. No design treatment beyond legibility.

### 3. JS — Parser Module (`src/js/parser.js`)

Create a dedicated ES module responsible for text parsing:

- **Export** a single function: `parseOccurrences(rawText)`.
- **Strategy**: Split the input on newlines (and/or common delimiters such as commas or semicolons). For each token, attempt to construct a valid `Date` object using `Date.parse()` / `new Date()`. Reject tokens that produce `NaN` or invalid dates.
- **Supported format heuristic**: The parser does not need to enumerate every format explicitly. It must handle, at minimum:
  - ISO 8601 (`2024-03-15T14:30:00`, `2024-03-15 14:30:00`)
  - Common locale-style dates with times (`15/03/2024 14:30`, `03-15-2024 14:30`)
  - Any format natively parseable by the JS `Date` constructor.
- **Return value**: An object `{ valid: Date[], invalid: string[] }` where `valid` is an array of `Date` instances successfully parsed and `invalid` is an array of the raw tokens that failed.
- All times must be stored normalised to **UTC ISO 8601 strings** (`date.toISOString()`).

### 4. JS — Data Service (`src/js/dataService.js`)

Create a dedicated ES module for all `localStorage` interactions:

- **`loadRecords()`**: reads and JSON-parses the `intervalTracker.records` key; returns an array of ISO 8601 strings (empty array if absent).
- **`saveRecords(records)`**: serialises and writes the full records array back to `localStorage`.
- **`addRecords(newDates)`**: loads existing records, merges `newDates` (ISO strings) deduplicating by exact string equality, sorts chronologically (ascending), and persists the result. Returns the final stored array.

### 5. JS — UI Controller (`src/js/uiController.js`)

Create a dedicated ES module that wires DOM events to the parser and data service:

- On `submit-occurrences` click:
  1. Read the raw text from `#occurrence-input`.
  2. Call `parseOccurrences(rawText)`.
  3. If `valid` is empty, display an error in `#input-feedback` listing the invalid tokens.
  4. If `valid` is non-empty, call `addRecords()` with the ISO strings, clear the textarea, and display a success message indicating how many records were added (and how many tokens were skipped, if any).
  5. Call `renderList()` to refresh the display.
- **`renderList()`**: loads all records from `dataService`, renders each ISO string as a plain `<li>` inside `#occurrence-list`. This is intentionally unstyled/minimal — definitive rendering is a future task.

### 6. JS — Entry Point (`src/js/script.js`)

Update `initializeApp()` to:

- Import and use `loadRecords` from `dataService.js` (replacing the inline `localStorage` call).
- Import `uiController.js` and call `renderList()` on startup so previously persisted records are shown immediately.

---

## Checklist of Final State

- `src/js/parser.js` exists and exports `parseOccurrences(rawText)` returning `{ valid, invalid }`.
- `src/js/dataService.js` exists and exports `loadRecords`, `saveRecords`, `addRecords`.
- `src/js/uiController.js` exists and handles submit events and initial list render.
- `src/index.html` contains `#occurrence-input`, `#submit-occurrences`, `#occurrence-list`, and `#input-feedback`.
- Submitting valid datetime text persists records to `localStorage` and updates the display.
- Submitting entirely invalid text shows a descriptive error and does **not** modify storage.
- Partially valid input persists the valid subset and reports skipped tokens.
- Duplicate entries (same ISO string) are not added twice.
- Records are displayed in chronological order.
- `src/css/styles.css` contains minimal structural rules for the new elements.

(Upon completion, subsequent tasks will build the definitive display layer and statistics engine on top of this persistence foundation.)
