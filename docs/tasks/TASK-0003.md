# TASK-0003: Unit Tests for TASK-0002 Business Logic

This task retroactively adds the unit test suite mandated by **DEC-0001** (see `docs/decisions.md`) for the two business-logic modules introduced in TASK-0002: `parser.js` and `dataService.js`.

Refer to `docs/testing.md` for the canonical toolchain setup, file conventions, and coverage expectations.

---

## Goal

Achieve ≥ 80 % line coverage on `src/js/parser.js` and `src/js/dataService.js` through focused unit tests that document and verify all significant behaviours.

---

## Steps

### 1. Toolchain Bootstrapping

- Initialise a `package.json` at the project root if one does not exist (`npm init -y`).
- Install dev dependencies as specified in `docs/testing.md`:

    "test:coverage": "vitest run --coverage"

tests/

### 3. `tests/unit/parser.test.js`

Write tests for every exported function in `src/js/parser.js`. Required cases:

**Empty / whitespace input**
- Blank string returns `{ valid: [], invalid: [] }`.
- String containing only whitespace returns the same.

**Single valid tokens — standard formats**
- ISO 8601 with `T` separator: `"2024-03-15T14:30:00"` → 1 valid Date.
- ISO 8601 with space separator: `"2024-03-15 14:30:00"` → 1 valid Date.
- Date-only string: `"2024-03-15"` → 1 valid Date (time defaults to midnight).
- Slash-separated with time: `"15/03/2024 14:30"` → 1 valid Date.
- Dot-separated with time: `"15.03.2024 14:30"` → 1 valid Date.

**Unambiguous day-first ordering**
- `"25/03/2024 09:00"` (day > 12, unambiguous) → parsed correctly as 25 March.

**Invalid tokens**
- A plain word `"not-a-date"` → appears in `invalid`, absent from `valid`.
- An empty token between delimiters is silently ignored (not counted as invalid).

**Multiple tokens — delimiters**
- Newline-separated list of 3 valid dates → `valid.length === 3`.
- Comma-separated list → same result.
- Semicolon-separated list → same result.
- Pipe-separated list → same result.

**Mixed valid and invalid**
- Input with 2 valid and 1 invalid token → `valid.length === 2`, `invalid.length === 1`.

**Returned type**
- Each element of `valid` is an instance of `Date`.
- `date.getTime()` is not `NaN` for any element of `valid`.

### 4. `tests/unit/dataService.test.js`

`dataService.js` depends on `localStorage`. Use Vitest's `jsdom` environment (already configured) which provides a `localStorage` stub. Reset storage in a `beforeEach` hook.

Write tests for all three exports:

**`loadRecords()`**
- Returns `[]` when `localStorage` is empty.
- Returns the stored array of ISO strings when data exists.
- Returns `[]` (does not throw) if the stored value is malformed JSON.

**`saveRecords(records)`**
- Persists the provided array so a subsequent `loadRecords()` returns the same values.
- Overwrites previously stored data.

**`addRecords(newDates)`**
- Adds new Date objects and returns the resulting ISO string array.
- Deduplicates: adding the same datetime twice results in only one entry.
- Sorts chronologically: inserting an earlier date into a list containing a later date produces ascending order.
- Merges with existing records: pre-existing entries are preserved.
- Returns the complete updated array.
